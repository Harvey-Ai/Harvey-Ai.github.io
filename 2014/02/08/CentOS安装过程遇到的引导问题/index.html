CentOS安装过程遇到的引导问题

2014-02-08 | System

寒假回来第一件事情就是给家里电脑更新换代，给家里电脑装了CentOS6，没想到整整搞了一个寒假，直到最后一天才安装完毕。

安装CentOS

家里没有Linux光盘，没有U盘，只能用硬盘安装的方法，我这里使用了修改grub引导的方法进行安装，原理很简单，电脑的头分区是一块叫mbr的扇区，一共512Byte，记录了启动的信息和分区表信息，修改这里并让电脑先引导grub便能引导linux安装程序。Fedora15以上版本的系统用的是OPT引导方式，家里机器安装不上，于是换更低的版本，没想到还搞不定，可能是分区格式是ntfs的缘故，最后选择装了CentOS，后来还使用了在线安装的方法。

第一次分区

安装好CentOS后，安装一些服务，配置了一下系统，并且修整了一下分区，用fdisk删除了一些多余的分区。没想到，删除后没法进入系统了，研究发现fdisk把分区名改了，但是并没有对grub进行相应的修改，原来/dev/sda10由于/dev/sda9被删除后被调整为/dev/sda9，而grub仍然使用(hd0, 9)作为启动分区，造成无法启动的错误。于是手动用grub进行引导：

  • root (hd0, 8)，
  • 然后调用configfile加载我的grub配置 /boot/grub/grub.config，修改grub里面的参数，把root (hd0,9)全部改成(hd0, 8)，然后成功进入系统。
  • 在系统下编辑/boot/grub/grub.conf文件，恢复错误的(hd0,9)。
  • 然后用grub，依次执行root(hd0, 8)，setup(hd0)重写mbr，再次进入系统，OK。

第二次分区

后来又一次调整分区的时候，忘记将新的grub重写入mbr，结果造成了莫名其妙的错误，开机显示error 22，这下算完完了，连grub都没法启动（这里感觉比较奇怪，为什么么fdisk不仅干修改disk table的事情，还修改bootloader，导致我的grub都不去），系统完全瘫痪了，只能靠其他介质进行引导。尝试用USB进行引导，家里电脑由于比较老旧不支持，没办法跑到街上去买一个grub工具，结果没买到。后来想了一个区曲线救国的方法，先装windows，在用在windows下装grub，然后重新进grub进行引导。

  • 安装好windows后，成功使用grub，然后敲root (hd0,5)，然后makeactive，这里始终说是not device requested，搞得很郁闷，用DOS工具diskgen都能查看linux文件系统下的文件，为什么总说是不合适的deivce呢，（分区是ext3格式的），用find也发现不到盘中的内容，然后用cat看能不能看到盘下的内容，竟然能cat到/boot/grub/grub.conf，这让我喜出望外，文件应该是没问题的。
  • 于是手工加载kernel和ramdisk，boot，不成功，看了一下错误可能和我的U盘有关，把掉U盘，然后调整了kernel加载的参数加上了ro root=/dev/sda6，然后boot，看着kernel进出了检测阶段，我的心里不知道有多高兴了，知道这下没问题了。进了系统以后，改了grub.conf和mbr之后，再次启动就没问题了。

两次现象分析

前后两次删除分区，为什么第一次grub能用，第二次grub不能用的呢？后来我想了一阵子，原因并不是fdisk搞坏了bootloader，fdisk并没有修改任何bootloader的内容，虽然没有证据。但是很可能还是出在分区上，因为MBR只存储了grub的stage1部分，stage2需要文件系统的支持，第二次删除分区不仅进行了删除操作还对剩余空间做了修改，这样可能导致找不到stag2文件的位置，所以grub不能正常运行。

备注 1. MBR：Master Boot Record，位于硬盘的第一柱面、第一磁头的第一个扇区上，和BIOS是一套启动装置。一共512个字节，bootloader + device table + magic number，分别占446, 64和2个字节，分区表一共有4项，所以MBR引导的系统最多4个主分区，支持最大硬盘容量是2T，linux中的stage1就是grub要写入的。 2. grub引导分2个阶段，第一阶段叫stage1，512个字节（正好是MBR的大小），因为mbr大小有限，stage1是初次引导，然后有stage1_5是stage1和stage2衔接部分（因为有不同的文件系统），接下来是真正的引导程序stage2。

3. 使用记法，/dev/sdaX中的X要比 (hd0, Y)中的Y大一。
